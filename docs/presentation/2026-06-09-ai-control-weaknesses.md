# AI 도구 활용의 단점과 통제 — 이 프로젝트 실제 사례

> 발표 핵심: "AI는 강력하지만 그대로 두면 벗어난다. **설계와 게이트를 사람이 쥐고 있을 때만** 통제된다."
> 아래는 전부 이 프로젝트(insurance / distribution)에서 **실제로 일어난 일**이며 git·코드·문서로 증거가 남아있다. 날조 없음.

---

## 0. 전제 — 무엇을 사람이 했고, 무엇을 AI가 했나

- **사람(나)**: 유스케이스 다이어그램(UC01~17), 클래스 다이어그램(도메인 모델), 단계적 진화 전략(TUI→DB→웹→Spring), 그리고 **모든 되돌리기 어려운 결정**(ADR).
- **AI**: 위 설계를 입력으로 받은 구현 노동, 테스트 작성, 리팩토링.
- 통제의 정의: *"AI가 도메인 모델·아키텍처를 벗어나려 할 때 사람이 만든 장치로 되돌리는 것."*

---

## 1. AI의 단점 ① — 성급한 추상화 (오버엔지니어링)

- **증상**: DB도 없는 콘솔 단계에서 AI가 컨트롤러 계층부터 분리(`3fc0ce3 feat: 컨트롤러 분리`). 아직 분리할 이유(분산·웹)가 없는데 "좋은 구조"를 미리 끌어다 씀.
- **발견**: 계층만 늘고 실익이 없음 — 데이터는 여전히 메모리, UI는 콘솔.
- **통제/해결**: `3e57d68 Revert "feat: 컨트롤러 분리"` 로 되돌림. **AI 제안을 사람이 명시적으로 기각한 git 증거.**
- **교훈**: AI는 "지금 필요한 구조"가 아니라 "교과서적 구조"를 제안하는 경향. 진화 단계에 맞는 복잡도는 사람이 통제한다.

## 2. AI의 단점 ② — 유스케이스 모델과 어긋난 구현

- **증상**: 자동차사고 보상심사 담당자 배정을, 의료청구처럼 **자동 배정**으로 구현. 자동차사고는 미배정 큐에서 사람이 골라 배정하는 흐름이 맞음.
- **발견**: grill-with-docs로 구현을 UC14/UC09 시나리오에 대고 취조하다 불일치 확인.
- **통제/해결**: `206bf25 자동배정→수동배정 변경`, 상태도 `23cc3b0 PENDING(접수)/IN_REVIEW(심사중) 분리`. ADR 0009로 결정 고정.
- **현재 코드**: 복잡 의료청구는 자동(`ClaimService` assignAutomatically), 자동차사고는 미배정 생성 후 수동(`CarAccidentService` + `POST /staff/claims/{id}/assign`).
- **교훈**: AI는 "비슷해 보이는 케이스"를 한 패턴으로 뭉개려 함. 유스케이스가 다르면 분기해야 한다는 건 도메인 문서가 강제한다.

## 3. AI의 단점 ③ — 레이어 패턴 일탈 (정직한 미완)

- **증상**: 거의 모든 컨트롤러는 `Controller → Service`만 호출하는데, `ProfileController` 한 곳만 컨트롤러가 `userRepository`를 직접 호출(레이어 건너뜀).
- **발견**: 도메인 전체 패턴 점검 중 1곳 예외 발견.
- **현재 상태**: 남아있음(사소한 조회라 동작엔 무해). **발표에서 숨기지 않고 "이런 일탈이 한 곳 있었고 점검으로 잡아냈다"고 정직하게 제시** → 일관성 통제가 실제로 작동함을 보여주는 증거로 사용.
- **교훈**: AI 산출물의 일관성은 자동으로 보장되지 않는다. 패턴 일탈은 사람의 주기적 점검으로만 잡힌다.

## 4. AI의 단점 ④ — 용어 표류 & 도메인 오염 (예방형 통제)

- **증상(일반)**: AI는 같은 개념을 account/user/policyholder처럼 섞어 쓰고, 도메인 클래스 안에 `System.out.println`(가입 성공 안내 등) 같은 출력/표현 로직을 넣으려는 경향.
- **통제 장치**:
  - `CONTEXT.md` 용어집을 단일 출처로 강제 → 동의어 혼용 금지(Policyholder로 정하면 user/account 금지).
  - ADR 0001 "다이어그램 메서드는 책임이지 배치가 아니다" → 비즈니스 흐름은 Service가 조율, 엔티티는 상태변경만.
- **결과(증거)**: insurance 백엔드 도메인에 `System.out` **0개**(grep). 엔티티는 JPA 어노테이션만, Spring/웹 어노테이션은 Service/Controller에만 격리. (대조군: distribution 단계는 `BenefitPayment.notifyPolicyholder`에 println이 남아있었음 → 웹 전환하며 제거.)
- **교훈**: 표류는 사후 수정보다 사전 규약(용어집·ADR)으로 막는 게 싸다.

## 5. AI의 단점 ⑤ — 검증 없이 코드부터 (TDD 게이트로 차단)

- **증상(일반)**: AI는 요청하면 프로덕션 코드를 즉시 쏟아냄. 테스트는 뒤로 미루거나 생략.
- **통제 장치**: 프로젝트 철칙 — *실패하는 테스트 없이 프로덕션 코드 금지*. Red→Verify Red→Green→Verify→Refactor. 태스크마다 서브에이전트 2단계 리뷰(spec 준수 → 코드 품질).
- **교훈**: AI 속도를 살리되 방향은 테스트가 잡는다.

---

## 6. 통제의 결정적 증거 — 다이어그램 ↔ 코드 추적성

내가 그린 다이어그램이 코드로 **그대로** 구현됐고, 의도적으로 벗어난 지점은 전부 ADR로
기록한 뒤 다이어그램에 역반영했다. 즉 변경이 "AI가 멋대로"가 아니라 "사람이 결정 → 문서가 따라옴".

| 변경 | 코드 현실 | 결정 근거 | 다이어그램 반영 |
|---|---|---|---|
| ClaimStatus 확장 | +COMPLETED, +FAILED | ADR 0007 (지급 결과 추적) | 04_claim / total 2-11 |
| ApplicationStatus 확장 | +CANCELLED | 신청 취소 흐름 | 03_contract / total 2-7 |
| 지급심사 대상 일반화 | `BenefitPaymentReview → Claim`(추상) | ADR 0009 (자동차사고도 진입) | 05_review / 00_overview / total |
| 배정 방식 분기 | 복잡의료=자동 / 자동차=수동 | `206bf25`, ADR 0009 | UC09/UC12/UC14 |
| AccidentRecord 미구현 | 집계값으로 갈음, @Embeddable 단순화 | 도메인 단순화 | 07_external / total 2-18 |

> 이 표(다이어그램 vs 코드 1:1 대조)를 슬라이드 한 장으로 보여주는 것이 발표에서 가장 강력하다.

---

## 7. 한 줄 결론

> "AI는 통제 가능하다. 단, **설계 문서·용어집·ADR·TDD라는 게이트가 사람 손에 있을 때만.**
> 이 프로젝트의 git 히스토리가 그 통제의 로그 그 자체다."
