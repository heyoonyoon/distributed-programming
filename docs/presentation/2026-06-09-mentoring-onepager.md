# 멘토링 1페이지 — 보험 시스템 / AI 통제

## 한 줄 주제
설계(클래스·유스케이스 다이어그램)는 사람이, 구현은 AI가. **통제의 증거는 git 히스토리.**

## 무엇을 만들었나
의료·자동차보험 가입~청구~지급 시스템. 액터 3(고객·직원·시스템).
핵심 흐름: 상품→가입신청→직원심사→**계약 자동생성**→납입 / 청구·사고→보상심사→지급.

## 진화 4단계 (의도된 분산화)
TUI(콘솔, 도메인만) → +DB(distribution, JDBC/DAO) → 웹/분산 → **Spring 마이그레이션(insurance)**.
각 단계 git 증거 존재(`8720dfa`→…→`af491b0`+Epic0~4). 단계가 바뀌어도 도메인 모델 불변.

## 아키텍처(항상 일정)
`웹 → Controller → Service → Domain → Repository → DB`, Entity→DTO는 `from()` 정적 팩토리.
도메인에 `System.out` 0개(grep). 엔티티는 JPA만, 웹 어노테이션은 상위 계층에 격리.

## AI 통제 장치 4
CONTEXT.md 용어집(동의어 차단) · ADR 9개(되돌리기 어려운 결정) · TDD 게이트 · 일정한 레이어 패턴.

## 단점(AI가 못한 것) → 해결 ★멘토링 핵심
1. 성급한 추상화: 컨트롤러 조기 분리(`3fc0ce3`) → **Revert**(`3e57d68`).
2. UC 불일치: 자동차 배정 자동(`bd2d7ce`) → **수동 정정**(`206bf25`).
3. 레이어 일탈: `ProfileController` Repository 직접 호출 → 점검으로 발견(정직히 인정).
4. 용어 표류·도메인 오염 → 용어집·ADR로 사전 차단.

## 다이어그램↔코드 정합성 (이번에 정리함)
ClaimStatus +COMPLETED/FAILED(ADR0007), ApplicationStatus +CANCELLED, 지급심사 대상=Claim 일반화(ADR0009),
AccidentRecord 미보유(집계 갈음). 모두 다이어그램에 역반영 완료.

## 데모 계획
1순위(진짜 동작): 가입신청→직원 조건부승인+할증→계약 자동생성→납입(역할 교차).
2순위(조건): 자동차 보상심사 화면 완성 시 접수→미배정 큐→수동배정→지급.
원칙: 목업(청구/사고/배정) 동작하는 척 금지.

## 발표 결론
"AI는 통제 가능하다 — 설계·용어집·ADR·TDD가 사람 손에 있을 때만. git이 그 통제의 로그다."
