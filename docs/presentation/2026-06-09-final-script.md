# 기말 발표 스크립트 (목표 10분) — 보험 시스템 / AI 통제

> 채점: 내용 70 + 발표 30(데모·AI활용도 비중 큼). 핵심 서사 = "설계는 사람, 구현은 AI, 통제의 증거는 git."
> 시간은 발표 10분 + 질문 여유. 실제로 10분 안에 끝나도록 소리내어 리허설할 것.
> ⚠️ 목업 화면(청구/사고/배정)은 동작하는 척 시연 금지. 라이브는 "진짜 도는 것"만.

---

## [0:00–0:30] 오프닝 (슬라이드 1: 타이틀)

"안녕하세요. 의료·자동차 **보험 가입부터 청구·지급까지**를 다루는 시스템입니다.
오늘 발표의 진짜 주제는 기능 자랑이 아니라 — **제가 이걸 AI로 만들면서, AI를 어떻게 통제했는가**입니다.
설계는 제가 했고, AI는 구현을 했습니다. 그 경계가 오늘의 이야기입니다."

## [0:30–2:00] 도메인 한눈에 (슬라이드 2: 도메인 흐름도 — 말+그림)

"먼저 뭘 만들었는지 30초만. (그림 가리키며)
상품 조회 → 가입 신청 → **직원 심사** → 승인되면 **계약 자동 생성** → 보험료 납입,
그리고 사고가 나면 청구·접수 → 보상 심사 → 지급. 액터는 셋입니다 — 고객, 보험사 직원, 그리고 자동 처리하는 시스템.
회원가입 같은 뻔한 건 건너뜁니다. 도메인이 살아있는 지점만 보여드릴게요."

## [2:00–3:30] 라이브 데모 (슬라이드 3: "DEMO" / 화면 전환)

> 데모 전 한 문장: "지금 보실 건 **고객이 가입 신청하고, 직원이 심사·승인하면 계약이 자동으로 만들어지는** 흐름입니다."

1. (고객) 상품 목록 → 하나 골라 **가입 신청**.
2. (직원 로그인) 심사 대기 목록 → 그 건 열기 → **조건부 승인 + 할증률 입력**.
3. 승인 즉시 **계약 자동 생성** → 고객 화면에서 계약/증권 확인.
4. (고객) 보험료 **납입**.

"여기서 포인트 하나. 제가 직원으로 넣은 **할증률이 그대로 계약 보험료에 반영**됩니다.
이건 제가 ADR로 못박은 도메인 규칙(조건부 승인은 심사에만, 할증은 최종 보험료에)이 코드로 살아있다는 뜻입니다.
화면이 도는 게 아니라 **규칙이 도는** 겁니다."

> (자동차 보상심사 프론트가 완성됐으면) "하나 더 — 자동차사고는 접수되면 미배정 큐에 쌓이고, 직원이 골라 배정한 뒤 지급액을 사정합니다." 짧게.
> (미완성이면) 언급만: "보상심사는 백엔드가 완성돼 있고 화면 연동은 진행 중입니다. 동작은 테스트로 보여드릴 수 있습니다." — 목업을 도는 척하지 않는다.

## [3:30–3:50] 주제 선언 (슬라이드 4)

"데모는 됩니다. 그런데 교수님이 보시려는 건 '됐냐'가 아니라 '어떻게 통제했냐'죠.
지금부터가 본론입니다."

## [3:50–5:30] 4단계 진화 — 설계는 사람, 구현만 AI (슬라이드 5: 진화 타임라인)

"저는 한 번에 스프링으로 가지 않았습니다. **일부러 4단계로 진화**시켰어요. (타임라인 가리키며)

1. **TUI 단계** — Scanner 콘솔, Main 한 파일에 도메인만. 컨트롤러·서비스 없음. (`8720dfa`)
2. **DB 단계** — `distribution` 프로젝트. JDBC + DAO로 H2 붙임. 여전히 콘솔. ORM 없음.
3. **웹/분산 단계** — REST 도입, 분산 프로그래밍 단계적으로.
4. **Spring 마이그레이션** — 지금의 `insurance`. Controller→Service→Repository(JPA). (`af491b0` 이후 Epic 0~4)

중요한 건 — 이 모든 단계에서 **클래스 다이어그램과 유스케이스 다이어그램은 제가 직접 그렸고, AI는 그걸 구현만** 했다는 겁니다.
그래서 단계가 바뀌어도 도메인 모델은 안 흔들렸어요."

## [5:30–7:30] 통제 메커니즘 + 아키텍처 (슬라이드 6: 레이어 패턴 / 슬라이드 7: 다이어그램↔코드 표)

"AI를 어떻게 통제했냐. 네 가지 장치입니다.

- **일정한 레이어 패턴**: 모든 도메인이 예외 없이 `웹 → Controller → Service → Domain → Repository → DB`,
  그리고 Entity는 항상 `XxxResponse.from(entity)` 정적 팩토리로 DTO가 됩니다. 패턴이 일정하니 AI 산출물도 일정해집니다.
- **CONTEXT.md 용어집**: 같은 개념을 user/account/policyholder로 섞어 쓰는 AI 버릇을 단일 용어로 강제했습니다.
- **ADR(결정 기록)**: 되돌리기 어려운 결정만 9개 기록. AI가 바꾼 게 아니라 제가 결정한 겁니다.
- **TDD 게이트**: 실패 테스트 없이 프로덕션 코드 금지.

결과 증거 하나 — 도메인 클래스에 `System.out.println`이 **0개**입니다(grep). 출력은 전부 웹 계층으로 넘겼어요.
(슬라이드 7) 그리고 이 표 — 제가 그린 **다이어그램과 실제 코드가 1:1로 맞습니다**.
의도적으로 벗어난 곳은 전부 ADR로 기록하고 다이어그램에 역반영했습니다."

## [7:30–9:30] 단점: AI가 삑사리 난 곳 → 어떻게 잡았나 (슬라이드 8: 발견→해결)

"이게 발표의 핵심입니다. AI가 못한 것들, 그리고 제가 어떻게 잡았는지.

1. **성급한 추상화**: AI가 DB도 없는데 컨트롤러부터 분리했어요(`3fc0ce3`). 실익이 없어서 제가 **되돌렸습니다**(`3e57d68 Revert`).
2. **유스케이스와 어긋난 구현**: 자동차사고 배정을 의료청구처럼 자동으로 했는데(`bd2d7ce`), UC상 자동차는 수동이 맞아서 **수동으로 정정**했습니다(`206bf25`).
3. **레이어 일탈(정직하게)**: 컨트롤러 하나(`ProfileController`)가 레이어를 건너뛰고 Repository를 직접 부릅니다. 점검하다 발견했고, 이런 일탈을 잡아내는 게 통제입니다.
4. **용어 표류·도메인 오염**: 용어집과 ADR로 사전에 막았습니다.

공통점 — AI는 빠르지만 방향은 못 잡습니다. **방향은 사람이 만든 게이트가 잡습니다.**"

## [9:30–10:00] 마무리 (슬라이드 9)

"정리하면 — AI는 통제 가능합니다. 단, **설계 문서·용어집·ADR·TDD가 사람 손에 있을 때만**요.
이 프로젝트의 git 히스토리가 그 통제의 로그 그 자체입니다. 감사합니다."

## 질문 대비 (Q&A 예상)
- "왜 청구 화면은 데모 안 했나?" → "프론트가 아직 목업이라 동작하는 척하지 않으려고요. 백엔드는 완성돼 테스트로 보여드릴 수 있습니다." (정직 = 가점)
- "AI가 짠 코드 비중은?" → "구현 라인은 대부분 AI, 설계·결정·리뷰는 전부 제가. 경계가 ADR/다이어그램에 남아있습니다."
- "Date/String id가 다이어그램과 코드가 다른데?" → "다이어그램은 분석 수준 개념 모델이라 의도적으로 유지했고, JPA 매핑(Long/LocalDate)은 구현 디테일로 봤습니다. 도메인 차이만 다이어그램에 반영했습니다."
