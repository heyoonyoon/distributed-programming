# 자동차사고도 BenefitPaymentReview를 거쳐 직원이 보험금을 사정한다

UC12(보험금 지급 심사)는 선행조건이 "UC05 의료보험의 복잡 청구건"으로 명시되어 있어, 자동차사고(CarAccidentReport)는 원래 접수·알림까지만 처리되고 심사·지급 흐름이 없었다. 그러나 자동차사고도 직원이 시스템 안에서 검토하고 보험금을 지급해야 한다는 제품 요구가 확인되어, 별도 심사 흐름을 새로 만드는 대신 **기존 BenefitPaymentReview를 일반화**해 자동차사고를 같은 보상심사 큐에 통합한다.

## 결정

1. `BenefitPaymentReview`의 청구 참조 타입을 `HealthInsuranceClaim`에서 부모 `Claim`으로 넓힌다. 의료 복잡건과 자동차사고가 같은 `/staff/benefit-reviews` 큐·배정·확정 흐름을 공유한다. 자동차사고는 접수 즉시 review가 생성·자동배정된다.
2. 자동차사고는 접수 시 청구금액이 없으므로(`requestAmount=0`), 승인 시 담당자가 지급 보험금을 직접 사정해 입력한다. 이 사정 금액은 별도 필드를 만들지 않고 `Claim.requestAmount`에 채운 뒤 기존 `BenefitPayoutService.pay()`가 그대로 지급에 사용한다. 의료건은 기존대로 청구 시 확정된 `requestAmount`를 쓴다.

## 트레이드오프

- 별도 `CarAccidentReview` 서브타입을 만들면 타입이 깔끔하게 분리되지만, 큐 엔드포인트가 두 서브타입을 합쳐야 하고 배정·지급 로직이 중복된다. 통합을 택해 기존 흐름 재사용을 극대화했다.
- `requestAmount` 한 필드가 의료에선 "가입자 청구금액", 자동차에선 "직원 사정 지급금액"이라는 두 의미를 갖는다. 의료도 청구금액 전액을 지급하므로 지급액 관점에서는 동작이 일관되며, 필드 분리에 따르는 매핑·마이그레이션 비용을 피하기 위해 오버로딩을 받아들였다.
