# 0004. 미납은 청구 회차 엔티티 없이 온더플라이로 계산한다

청구 회차(Installment)를 별도 엔티티/테이블로 영속화하지 않는다. 미납은 계약 정보로 매번 계산해서 도출한다.

- **청구 회차**: 계약 `startDate`부터 매월 같은 일자에 `monthlyPremium` 청구. n번째 기한 = `startDate + n개월`.
- **발생 회차 수**(asOf=오늘) = 기한이 오늘 이하인 회차 개수.
- **미납 회차 수** = 발생 회차 수 − `Payment(SUCCESS)` 건수 (음수면 0).
- **FIFO 충당**: 성공 납부는 가장 오래된 회차부터 채운 것으로 간주. Payment를 특정 회차에 매핑하지 않는다.
- **연체일수** = 가장 오래된 미납 회차 기한 ~ 오늘. **미납 원금** = 미납 회차 수 × monthlyPremium.

클래스 다이어그램(03_contract)에는 `Payment`와 `Notice`만 있고 회차 엔티티가 없다. 이 결정은 다이어그램을 그대로 유지한다.

## Considered Options

- **채택(온더플라이)**: 도메인 단순, 다이어그램 일치, "어느 회차를 냈나" 상태를 안 들고 다님. 단점: 부분 납부·회차별 상태를 표현 못 함(현재 범위 밖이라 무방).
- **Installment 엔티티 추가**: 회차별 dueDate/금액/납부여부를 영속화해 명시적. 단점: 다이어그램에 없는 클래스 추가, 회차 생성/동기화 로직 필요, 현 범위 대비 과설계.

되돌리기 어려움: 미납 판정이 이 계산 규칙에 의존하므로 나중에 부분납부·회차 상태가 필요해지면 데이터 모델을 바꿔야 한다.
