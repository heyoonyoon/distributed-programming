# 유스케이스 구현 워크플로우

## 원칙

유스케이스는 **반드시 하나씩** 구현하고, 구현 후 **반드시 수기 테스트**한다.
수기 테스트 통과 확인 전까지 다음 유스케이스로 넘어가지 않는다.

## 단계별 절차

### STEP 1: 시나리오 학습
```
사용자가 유스케이스를 지정하면:
1. @docs/usecases/UC0N_[이름].md 파일을 읽는다
2. @docs/class_diagram.md 파일을 읽는다
3. 읽은 내용을 바탕으로 구현 계획을 사용자에게 먼저 보고한다
4. 사용자 승인 후 구현 시작
```

### STEP 2: 구현 체크
구현 전 스스로 확인:
- [ ] 이 유스케이스에 필요한 클래스가 다이어그램에 있는가?
- [ ] 시나리오의 모든 step이 코드에 반영되어 있는가?
- [ ] 시나리오 flow와 코드 흐름이 동일한가?
- [ ] 설계에 없는 것을 추가하지 않았는가?
- [ ] `main()`의 switch case 번호가 유스케이스 순서와 일치하는가?

### STEP 3: 구현 완료 후 보고
구현이 끝나면 Claude는 반드시 다음 형식으로 보고한다:

```
=== 구현 완료 보고: [유스케이스명] ===

[구현한 클래스/메서드]
- 클래스명: [이름] (다이어그램 N번)
- 추가 메서드: [목록]

[시나리오 flow 매핑]
- 시나리오 1단계 → [코드 메서드명]
- 시나리오 2단계 → [코드 메서드명]
- ...

[수기 테스트 체크리스트]
아래 순서대로 직접 실행하여 확인하세요:
1. 프로그램 실행: [실행 명령]
2. 메뉴에서 [N번] 입력
3. [입력값 1] 입력 → 예상 결과: [결과]
4. [입력값 2] 입력 → 예상 결과: [결과]
5. ...

위 항목을 모두 확인하셨으면 다음 유스케이스 진행을 알려주세요.
===
```

### STEP 4: 테스트 결과 확인
- 사용자가 테스트 완료를 알리면 다음 유스케이스로 진행
- 테스트 실패 시 실패한 step만 수정 (다른 것 건드리지 않음)

## 유스케이스 진행 상태 추적

Claude는 세션 시작 시 아래 상태를 확인한다:

| UC # | 유스케이스명 | 구현 | 테스트 |
|------|------------|------|--------|
| UC01 | (레포트 기반 입력) | ⬜ | ⬜ |
| UC02 | (레포트 기반 입력) | ⬜ | ⬜ |
| ...  | ...         | ⬜ | ⬜ |

✅ = 완료 / ⬜ = 미완료

## 흐름 변경 금지 예시

```
시나리오: 사용자가 이름 입력 → 시스템이 중복 확인 → 중복이면 오류 출력 → 다시 입력

금지: 사용자가 이름 입력 → 저장 (중복 확인 생략)
금지: 중복 확인 전에 저장부터 하는 것
금지: 오류 대신 예외 던지는 것 (설계에 없으면)

허용: "이름을 입력하세요:" → "이미 존재하는 이름입니다." → "다시 입력하세요:"
```
